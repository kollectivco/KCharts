# Kontentainment Charts - Phase 5 Packaging & Release Guidelines

To ensure the GitHub-powered updater functions reliably in production, all releases MUST follow these strict packaging and identity requirements.

## 1. Release Identification
*   **Version Tag**: Releases must be tagged with a version string matching the `AMC_PLUGIN_VERSION` constant (e.g., `4.3.8`).
*   **Target Repository**: `kollectivco/KCharts`
*   **Main Plugin File**: `kontentainment-charts.php`
*   **Plugin Slug**: `kontentainment-charts`

## 2. Packaging Specification
The updater is "asset-strict". It will NOT offer an update unless it finds an official ZIP asset attached to the GitHub release.

*   **Asset Name**: `kontentainment-charts.zip`
*   **Internal Structure**:
    *   The ZIP root MUST be a single folder named `kontentainment-charts/`.
    *   The main entry point `kontentainment-charts.php` MUST be at the root of that folder.
    *   Example: `kontentainment-charts.zip` -> `kontentainment-charts/kontentainment-charts.php`.

> [!WARNING]
> DO NOT rely on GitHub's autogenerated "Source code (zip)" buttons. These zips use inconsistent folder naming (e.g., `KCharts-v4.3.8/`) which can break the WordPress plugin pathing and cause deactivation after update.

## 3. Pre-Release Checklist
Before pushing a new production release:
1.  **[ ] Version Sync**: Verify `Version` in `kontentainment-charts.php` header, `AMC_PLUGIN_VERSION` constant, and `Stable tag` in `readme.txt` all match exactly.
2.  **[ ] Asset Build**: Run the build script (if any) to generate the production-ready `kontentainment-charts.zip`.
3.  **[ ] Asset Verification**: Verify the ZIP contains only the `kontentainment-charts/` folder and no `.git`, `.github`, or local dev artifacts.
4.  **[ ] GitHub Draft**: Create a new GitHub Release, upload the `kontentainment-charts.zip` asset, and then publish (do not leave as draft).

## 4. Maintenance & Safety
*   **Identity Locking**: The `AMC_PLUGIN_SLUG` constant is the "source of truth" for the folder name. Any change to this slug requires a manual migration of the `WP_PLUGIN_DIR` folder on existing sites.
*   **Updater Resilience**: If the GitHub API is down, the plugin remains fully operational. The updater fails gracefully into a silent state until the API or cache expires.
*   **Manual Override**: Operators can always force an update check via the "Check for updates" link in the WordPress Plugins screen.
